# 小五 FixTask 真实闭环修复报告

日期：2026-06-19

## 背景

本次修复针对“小五第一次验收不通过后流程僵持”的问题。用户要求：

- 使用真实数据、真实流程。
- 不允许 mock、补丁式伪装结果或让流程看起来通过。
- 读取并遵守根目录 `AGENTS.md` 中的“真实能力与开发条件不足处理规则”。

已读取该章节，并按其中规则处理：真实调用失败时暴露失败；真实能力未达成时不伪装成完成；产品内 LLM 产物必须来自产品配置的 LLM provider；流程通过必须有真实证据。

## 判断

僵持原因主要不是 CC 没有能力执行，而是小五的流程制度缺口：

1. 第一次 ReviewResult 判定不通过后，UI 和后端没有正式的“整改任务”产物。
2. 小五没有把不合格项转成新的、可执行的 CC 指令。
3. 后续也没有第二次 ImplementationReport 和最终复验的状态机入口。

因此，CC 只能停在“我已经提交了上一轮报告”的状态。要继续，必须由小五发出明确的下一步任务，而不是让 CC 自行猜测。

## 本次修复

### 1. 新增 FixTask 阶段

新增接口：

- `POST /api/xiaowu/fix-task`

触发条件：

- 已有 PRD。
- 已有第一轮 TaskSpec。
- 已有第一轮 CC ImplementationReport。
- 已有小五第一次 ReviewResult，且流程进入不通过状态。

行为：

- 小五通过真实 LLM provider 生成 `FixTask`。
- 写入真实产物 `fixtask-smallcalc-v1`。
- 写入 `.agentbus/cc-inbox/task-smallcalc-implementation-001.json`。
- 通过 Codex App Server 调用器启动第二轮 CC。

### 2. CC 第二轮必须真实实现

第二轮 TaskSpec 的约束从：

```json
"doNotImplementSmallCalcYet": true
```

变为：

```json
"doNotImplementSmallCalcYet": false
```

CC prompt 也改为读取 TaskSpec 后按目标仓库执行。如果 TaskSpec 要求实现 SmallCalc，就必须真实创建或修改代码、运行验证命令，并写回真实 ImplementationReport。

### 3. 新增最终验收阶段

新增接口：

- `POST /api/xiaowu/final-review`

触发条件：

- 已有 FixTask。
- 已有第二轮 CC ImplementationReport。

行为：

- 小五后端读取目标仓库文件快照。
- 小五后端独立执行：
  - `npm test`
  - `npm run build`
- 将 FixTask、CC 第二轮报告、后端复验结果、仓库文件快照一起交给真实 LLM 生成最终验收报告。
- 写入真实产物 `review-smallcalc-final-v1`。

### 4. UI 状态机修正

主按钮从固定步骤变为真实状态驱动：

- `小五创建 PRD`
- `发送 TaskSpec 给 CC`
- `等待 CC 报告`
- `小五验收报告`
- `要求 CC 实现 SmallCalc`
- `等待 CC 实现报告`
- `小五再次验收`
- `流程已完成`

等待 CC 报告时按钮不可点击，避免用点击推进伪造状态。

## 真实发现的问题

最终验收第一次尝试时，小五没有独立复验目标仓库，只依据报告和有限文件快照，LLM 给出了“不通过/不可判定为通过”。这不是坏事，反而暴露了制度缺口：验收不能只看 CC 报告。

因此本次追加了后端独立复验机制：最终验收前必须由小五后端实际运行目标仓库验证命令，并把命令输出作为验收依据。

## 剩余风险

SmallCalc 目标仓库当前是真实代码，但 git 制度还未完成：

- 当前目标仓库是无提交状态。
- 当前分支为 `master`。
- 原 TaskSpec 中提到的 `baseBranch main` 和 `workBranch cc/smallcalc-mvp` 尚未落实。
- 还没有真实 GitHub PR 创建动作。

这不影响本次“PM Agent -> CC -> 报告 -> 验收 -> FixTask -> CC 修复 -> 再验收”的本地真实闭环验证，但会影响后续进入真实 GitHub PR 工作流。

## 建议下一步

下一步应把 git 和 PR 制度纳入小五的 TaskSpec：

- 小五创建任务时明确目标仓库、base branch、work branch。
- CC 启动前由调用器确认或创建工作分支。
- CC 完成后提交 commit。
- 小五验收时检查 commit、diff、测试、构建。
- 通过后再进入 GitHub PR 创建或更新。
