# 小五工作台 UI 设计

## 目标

本次设计目标是把“小五 PM Agent 管理 CC Coding Agent 完成 SmallCalc MVP”的过程转成一个可视化、可重放的 UI 工作台。用户可以在界面中一步步推进流程，看到每一步的角色、状态、产物、证据和结果。

## 范围

本阶段实现的是小五和 CC 的真实通信探针。界面默认从 `0 / 7` 的等待状态开始，SmallCalc 程序不会提前存在；只有在用户点击“发送 TaskSpec 给 CC”后，小五才会把真实任务写入 `.agentbus/cc-inbox/`，本地 orchestrator 再拉起真实 `codex exec`。

## 7 步流程

1. 小五创建 SmallCalc PRD。
2. 小五给 CC 安排 MVP 任务。
3. CC 完成第一次实现并提交报告。
4. 小五第一次验收。
5. 小五判定不通过并列出不合格项。
6. CC 根据验收结果修复并再次提交。
7. 小五再次验收并通过。

## UI 结构

- 顶部栏：展示小五工作台名称、流程控制按钮。
- 状态条：展示当前步骤、当前角色、流程进度和验收状态；未发令前显示等待小五发令。
- 左侧流程区：展示 7 步时间线，支持点击跳转。
- 中央详情区：展示当前步骤的说明、产物、证据和结果。
- 右侧通信区：展示小五和 CC 之间的消息包，包括消息 ID、消息类型、发送方、接收方、通信通道、状态和 payload。

## 通信表达

小五和 CC 的通信在 UI 中以消息包展示：

- `MSG-001`：小五通过真实 `.agentbus/cc-inbox` 把 TaskSpec 发送给 CC。
- `MSG-002`：CC 通过真实 `.agentbus/xiaowu-inbox` 把 ImplementationReport 回传给小五。
- `MSG-003`：小五通过 `ReviewRequest` 把不通过结论和不合格项发回 CC。
- `MSG-004`：CC 通过 `FixReport` 提交修复报告。
- `MSG-005`：小五通过 `Approval` 发出最终通过结论。

未到对应步骤时，消息不会出现；未发令前通信区显示“尚未通信”。CC 执行过程通过 `.agentbus/runs/<runId>/events.jsonl` 展示，包括 stdout、stderr 和退出码。

## 布局约束

该界面属于生产工具型 UI，采用桌面式布局。外层应用壳固定为浏览器可视窗口高度，主工作区内部滚动，不依赖 `body` 或 `main` 的页面级滚动承载内容。

## 后续替换方向

当前已接入真实 agentbus 通信和真实 Codex worker 探针，但还没有让 CC 实现 SmallCalc。后续正式版本应在小五验收通信链路后，再把 TaskSpec 升级为真实 SmallCalc 实现任务，并接入 GitHub Issue、PR、review、测试结果和小五验收结论。
